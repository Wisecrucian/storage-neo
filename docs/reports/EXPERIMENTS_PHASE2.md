## Exp1 — Read degradation curve vs write throughput

**Гипотеза:** при росте фона записи latency point-read и OLAP-read растёт из-за конкуренции за CPU/IO и фоновых merge; число активных parts и эффективное сжатие могут меняться.

**Метод:**

1. Калибровка: 1–4 потока писателей, 40 с каждый, batch=2000.
2. Для уровней 0 / 5k / 10k / 15k / 20k: выбран ближайший по калибровке `threads` (см. ограничение выше).
3. Писатели работают **во время** замера: стабилизация 35 с, измерение фактического RPS за 10 с, затем **без остановки писателей** — серия point-read и OLAP-read; снимок `active_parts` и compression ratio по `system.parts`.

**Результат**

| Target rec/s | Writer threads | Achieved write rec/s | Point read p50 / p95 / p99 (ms) | OLAP read p50 / p95 (ms) | Active parts | Ratio |
|---:|---:|---:|---|---|---:|---:|
| 0 | 0 | 0 | 30.0 / 71.0 / 84.6 | 192.5 / 267.7 | 29 | 9.150 |
| 5000 | 1 | 15400 | 64.3 / 91.3 / 97.8 | 446.3 / 864.9 | 25 | 9.049 |
| 10000 | 1 | 15200 | 101.5 / 221.0 / 849.9 | 633.6 / 916.3 | 22 | 8.883 |
| 15000 | 1 | 15600 | 66.2 / 130.2 / 133.0 | 486.6 / 781.4 | 30 | 8.778 |
| 20000 | 3 | 21000 | 85.0 / 148.8 / 154.2 | 419.9 / 500.8 | 31 | 8.656 |

**Вывод:** при ненулевой записи OLAP-read заметно дороже, чем в idle (p50 ~192 ms → до ~634 ms на промежуточном уровне). Point-read тоже растёт относительно тишины, но менее монотонно — влияет кэш, конкуренция и выборка по «горячему» `user_id`. `compression_ratio` плавно снижается с ростом объёма и фоновых merge — типичное поведение при интенсивном ingest.

## Exp2 — Parts explosion / sustained write + recovery

**Гипотеза:** при непрерывной высокой записи число активных **parts** растёт; после остановки записи merge постепенно снижает parts и может улучшить ratio.

**Метод:** 4 writer-потока × **5 минут**; каждые **10 с** снимок: `active_parts`, `compression_ratio`, `rows`, счётчик принятых записей API. Затем **2 мин idle** с тем же интервалом.

**Результат (сжатая выборка)**

| Фаза | Время | Active parts (типично) | Rows (макс из снимка) | Ratio (мин..макс) |
|------|--------|------------------------|------------------------|-------------------|
| Write | старт | 51 | ~38.0M | ~8.51 |
| Write | пик | **до 61** | — | плавно ↓ ~8.31 |
| Write | конец | 51 | ~40.6M | ~8.26 |
| Idle | через ~2 мин | **25** | ~43.0M | **~8.00** (плато) |

Полные ~30 точек — в `experiments_phase2_out.json` → `exp2_parts_timeline`.

#### Почему `rows` иногда «замирает» между снимками

`sum(rows)` из `system.parts` обновляется при появлении новых частей; между тиками merge/insert могут компенсировать друг друга — это нормальная гранулярность наблюдения.

**Вывод:** под длительной нагрузкой parts выросли до **~60+**, после паузы merge сократил их до **~25**. Ratio в idle стабилизировался около **8×** (по сравнению с ~8.5× в начале write-фазы) — интерпретация: больше «свежих» мелких частей → хуже глобальное сжатие, merge выравнивает.

## Exp3 — Projection cost on insert

**Гипотеза:** поддержка проекции `by_record_type` добавляет работу на пути insert/merge и может снизить устойчивый ingest throughput.

**Метод:** 30 с запись (4 потока, batch=2000) **с проекцией** → `ALTER TABLE ... DROP PROJECTION` → ещё 30 с → `ADD PROJECTION` + `MATERIALIZE PROJECTION` (восстановление).

**Результат**

| Режим | Throughput (rec/s) | Записано строк (HTTP OK) |
|--------|-------------------:|------------------------:|
| С проекцией | **20 805** | 632 000 |
| Без проекции | **20 241** | 614 000 |

**Вывод:** в данном прогоне проекция дала **~+2.8%** к throughput (разница в пределах вариативности нагрузки/OS). Для вывода «проекция дороже/дешевле на insert» нужны **повторения** (3–5 прогонов) и контроль фоновых merge. Качественно ожидается: проекция увеличивает **диск** и **CPU merge**; см. также Phase1/отдельный замер price/ benefit для read.

## Exp4 — Map vs materialized columns

**Гипотеза:** для частых измерений выгоднее физическая колонка (materialized), чем извлечение из `Map` на каждом запросе.

**Метод:** по **10** прогонов каждого запроса; добавлена `attr_source LowCardinality MATERIALIZED attrs['source']` (если ещё не было).

**Результат (ms)**

| Сценарий | Map / выражение | Materialized | avg (map vs mat) |
|----------|-----------------|--------------|-----------------:|
| GROUP BY lang | `attrs['lang']` | `attr_lang` | **760 vs 79** |
| GROUP BY source | `attrs['source']` | `attr_source` | **657 vs 430** |
| Фильтр lang='ru' | `attrs['lang']='ru'` | `attr_lang='ru'` | **609 vs 75** |

**Вывод:** для **lang** и фильтра по **lang** выигрыш materialized-колонки **крупный (~8–10×)**. Для **source** выигрыш есть, но скромнее — домен всего из **3** значений, планировщик и размер выборки играют роль; materialized всё равно снижает среднее время.

## Exp5 — Bloom filter (`chat_id`)

**Гипотеза:** bloom skip index уменьшает число читаемых гранул для `WHERE chat_id = const` и ускоряет запрос.

**Метод:** выбран наиболее частый `chat_id`; 10× замер latency; `EXPLAIN indexes=1`; `DROP INDEX` → повтор → восстановление + `MATERIALIZE INDEX`.

**Результат**

| Режим | avg / p50 / p95 (ms) | Granules (EXPLAIN, Skip / всего) |
|--------|----------------------|----------------------------------|
| С bloom | 64.3 / 62.5 / 84.4 | **2830 / 5256** |
| Без bloom | 57.6 / 54.7 / 77.3 | 5256 / 5256 |

**Вывод:** индекс **сократил гранулы ~46%**, но **latency не улучшил** (даже чуть хуже). Интерпретация: на таблице ~43M строк и тёплом кэше стоимость проверки skip-блоков + ветвление плана может сравняться с выгодой; для «холодных» данных / больших сканов выгода обычно заметнее. Рекомендация: фиксировать **read_bytes** из `system.query_log` при повторении.

## Exp6 — Kafka path vs прямой INSERT в ClickHouse

**Гипотеза:** прямой bulk `INSERT ... SELECT` в MergeTree даёт более высокий «сырой» RPS, чем путь API→Kafka→MV (разная семантика backpressure и форматы).

**Метод:**

- **Kafka path:** 30 с, 4 потока, batch=2000, Java API.
- **Direct:** 30 с непрерывных батчей `INSERT INTO events_raw SELECT ... FROM numbers(120000)` (нерепрезентативно для продукта, но хороший верхний ориентир для «чистого ClickHouse»).

**Результат**

| Путь | Throughput (rec/s) | Примечание |
|------|-------------------:|------------|
| Java → Kafka → CH | **19 012** | продуктовый контур |
| HTTP INSERT SELECT | **~380 000** | CH-native, без сериализации JSON в приложении |

**Вывод:** прямой INSERT **на порядки** быстрее — это ожидаемо: нет JSON/Kafka/producer, один широкий конвейер в CH. Практический смысл Kafka-пути не в максимальном RPS на лабораторной вставке, а в **буферизации, обратном давлении и отвязке** latency API от тяжёлого merge в CH.
